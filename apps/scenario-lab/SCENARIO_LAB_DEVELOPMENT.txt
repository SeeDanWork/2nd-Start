1. WHAT IS SCENARIO LAB?
================================================================================

Scenario Lab is a standalone Next.js application for testing and validating
ADCP's custody scheduling system. It simulates realistic co-parenting
conversations, disruptions, and schedule negotiations without requiring
the full backend infrastructure (NestJS API, PostgreSQL, Python optimizer).

PURPOSE:
  - Test the conversation flow end-to-end
  - Validate persona behavior under different scenarios
  - Stress-test the disruption/mediation pipeline
  - Generate deterministic schedule outputs
  - Produce diagnostic data for system tuning

Scenario Lab runs entirely in-browser + Next.js server. State is held
in-memory (no database). Each scenario is isolated and disposable.

--- behavior-engine.ts (800+ lines) ---
The brain of persona behavior. Contains:
  - Decision engine (evaluateProposal, evaluateDisruption)
  - Archetype-aware evaluation
  - Persona message generation (template library)
  - System message classification (keyword matching)
  - Topic classification (compound + single)
  - Reactive answer generation
  - Compound answer generation
  - Synthetic onboarding system (8-step flow)
  - Simulation metrics computation
  - Response timing calculations

--- disruption-engine.ts ---
Immutable state machine for disruption lifecycle.
8 states, pure transition functions.
Duplicate detection, event classification.
See CONVERSATION_LOGIC.txt Section 8.

--- explanation-engine.ts (355 lines) ---
Deterministic explanation traces. Three detail levels.
Score computation: fairness, transition, routine, fragmentation penalties.
Factor analysis: improved/unchanged/degraded per metric.
Personalized day summaries with recipient-aware pronouns.

--- message-router.ts (308 lines) ---
Asymmetric message routing. 10 message types.
Each parent gets contextually appropriate messages.
See CONVERSATION_LOGIC.txt Section 9.

--- operational-messages.ts ---
Day-to-day operational message generation.
11 message types from SILENT to FRICTION_DETECTION.
Schedule metrics computation (fairness, transitions, stability).
See CONVERSATION_LOGIC.txt Section 10.

--- personas.ts ---
6 parent persona definitions with behavior parameters.
Interaction archetypes for persona pairs.
See Section 9 below.

--- proposal-generator.ts ---
Generates coverage options for disruptions.
Options: full transfer, partial day, shortened coverage.
Fairness impact calculation, compensation day scheduling.

--- scenarios.ts ---
40+ scenario catalog entries across 11 categories.
5 difficulty levels. Response patterns for archetype-specific behavior.

--- schedule-generator.ts ---
Template-based schedule generation with locked night overrides.
5 patterns: alternating_weeks, 2-2-3, 3-4-4-3, 5-2, every_other_weekend.
Starts from next Monday, generates N weeks of daily assignments.

--- conversation/orchestrator.ts ---
Single entry point. 7 actions. Error normalization.
See CONVERSATION_LOGIC.txt Section 2.

--- conversation/state.ts ---
Pure state tracking functions.
See CONVERSATION_LOGIC.txt Section 4.

--- conversation/handlers/ (7 files) ---
One handler per action. Typed inputs/outputs.
See CONVERSATION_LOGIC.txt Section 3.

--- llm/client.ts ---
3 functions, all return null on failure.
See CONVERSATION_LOGIC.txt Section 7.

--- monte-carlo/ (4 files) ---
Statistical simulation runner. Runs N iterations of a scenario
with randomized persona decisions. Produces distribution of outcomes:
acceptance rates, conflict escalations, fairness drift.

6. API ROUTES
================================================================================

POST /api/simulate
  Body: { scenarioId, phone, action, body? }
  Actions: connect, send, auto_respond, simulate_pair, run_setup,
           step_day, inject_disruption
  Routes through orchestrator. Returns action-specific data.

GET /api/scenarios
  Returns all scenarios sorted by creation date (newest first).

POST /api/scenarios
  Body: ScenarioConfig
  Creates new scenario with 'draft' status. Returns full Scenario.

GET /api/scenarios/[id]
  Returns single scenario by ID.

PATCH /api/scenarios/[id]
  Body: Partial<Scenario>
  Updates scenario fields. Returns updated Scenario.

DELETE /api/scenarios/[id]
  Deletes scenario. Returns { success: true }.

POST /api/export
  Body: { scenarioId, type }
  Types: summary, conversation, schedule, diagnostics
  Returns formatted text export.

POST /api/monte-carlo
  Body: { config, iterations, seed? }
  Runs Monte Carlo simulation. Returns statistical results.

9. PERSONAS (6 Archetypes)
================================================================================

Each persona has behavior parameters on 1-5 scales:

1. COOPERATIVE ORGANIZER
   conflict: 1 | fairness: 3 | rigidity: 4 | tolerance: 4
   speed: fast | gaming: 0% | acceptance: 80%
   Prioritizes stability. Accepts reasonable proposals quickly.

2. FAIRNESS SCOREKEEPER
   conflict: 3 | fairness: 5 | rigidity: 3 | tolerance: 2
   speed: medium | gaming: 10% | acceptance: 50%
   Tracks balance carefully. Resists perceived imbalance.

3. FLEXIBLE BUT DISORGANIZED
   conflict: 1 | fairness: 2 | rigidity: 1 | tolerance: 5
   speed: slow | gaming: 0% | acceptance: 90%
   Accepts most changes but frequently causes disruptions.

4. STRATEGIC GAMER
   conflict: 4 | fairness: 3 | rigidity: 3 | tolerance: 2
   speed: medium | gaming: 70% | acceptance: 30%
   Exploits system rules for advantage. Injects extra time requests.

5. AVOIDANT PARENT
   conflict: 2 | fairness: 2 | rigidity: 2 | tolerance: 3
   speed: very_slow | gaming: 5% | acceptance: 60%
   Slow to respond. 40% chance of ignoring messages entirely.

6. HIGH CONFLICT CONTROLLER
   conflict: 5 | fairness: 4 | rigidity: 5 | tolerance: 1
   speed: fast | gaming: 30% | acceptance: 20%
   Rejects most proposals. Rigid and combative.

INTERACTION ARCHETYPES:
  Define expected dynamics for persona pairs. Include:
  - conflict_probability
  - typical resolution patterns
  - expected escalation paths

11. SCHEDULE TEMPLATES (5 Patterns)
================================================================================

Each template defines a repeating cycle of parent assignments:
(true = parent_a, false = parent_b)

ALTERNATING WEEKS (50/50):
  Week 1: A A A A A A A
  Week 2: B B B B B B B

2-2-3 (50/50):
  Week 1: A A B B A A A
  Week 2: B B A A B B B

3-4-4-3 (50/50):
  Week 1: A A A B B B B
  Week 2: A A A A B B B

5-2 (71/29):
  Every week: A A A A A B B

EVERY OTHER WEEKEND (79/21):
  Week 1: A A A A A A A
  Week 2: A A A A A B B

Schedule generation:
  - Starts from next Monday
  - Applies template pattern cyclically
  - Overrides with locked nights (specific days always with specific parent)
  - Marks transition days (custody switches)
  - Default: 8 weeks (56 days)

13. MONTE CARLO SIMULATION
================================================================================

Files: lib/monte-carlo/

Runs N iterations of a scenario with randomized persona decisions.
Each iteration:
  1. Create scenario from config
  2. Run setup (automated onboarding)
  3. Step through simulation weeks
  4. Inject disruptions at configured points
  5. Record all decisions, metrics, outcomes

Output:
  - Acceptance rate distribution
  - Conflict escalation frequency
  - Fairness drift over time
  - Gaming attempt detection rate
  - Schedule volatility metrics

UI: /monte-carlo page with scenario picker and results visualization.

15. KNOWN LIMITATIONS
================================================================================

1. IN-MEMORY STATE: All data lost on server restart. No persistence layer.
   Intentional for rapid iteration. Not suitable for production.

2. NO REAL LLM INTEGRATION: LLM router is optional and fire-and-forget.
   The system works fully without it. LLM adds intent classification
   and entity extraction as enrichment only.

3. SINGLE-USER: No authentication. No multi-user support. The store is
   shared across all browser sessions hitting the same server.

4. SCHEDULE GENERATION IS SIMPLIFIED: Uses template patterns + locked
   nights. Does not call the Python CP-SAT optimizer. Adequate for
   testing conversation logic but not production-quality schedules.

5. NO REAL-TIME: Step-day is manual. No cron or automated time progression.
   The user must explicitly advance the simulation day by day or in batches.

6. NO SMS/WHATSAPP: Phone simulators are UI-only. No actual message delivery.
   The phone numbers are identifiers, not real phone numbers.

7. RESPONSE TIMING: Response delays are calculated but not enforced.
   All responses are instant in the UI. Delay values are logged for analysis.

8. PERSONA RANDOMNESS: Decision outcomes have a random component (Math.random).
   The same scenario may produce different decisions across runs.
   Monte Carlo simulation addresses this via statistical aggregation.

9. PARENT B ONBOARDING: Parent B's automated onboarding is simplified.
   It may exit early if the compound answer generation gets stuck.
   A fallback ensures the "schedule created" message is always delivered.

10. SWC LOCKFILE WARNING: Next.js 15.5 shows "Found lockfile missing swc
    dependencies" on startup in monorepo context. Harmless — the server
    starts and runs correctly despite the error messages.

16. KEY DESIGN DECISIONS
================================================================================

1. DETERMINISTIC OVER LLM:
   Every piece of conversation logic can run without an LLM. The behavior
   engine uses numeric scoring formulas, not natural language generation.
   This makes the system testable, reproducible, and debuggable.

2. ASYMMETRIC MESSAGING:
   Each parent gets different messages based on their role in the current
   interaction. The reporting parent sees confirmation + duration questions.
   The other parent sees coverage requests + proposals. This mirrors how
   a real mediation system would work — each party has their own view.

3. IMMUTABLE STATE TRANSITIONS:
   The disruption state machine uses pure functions that return new objects.
   Each transition is a function call: createDisruption → setDuration →
   attachProposals → selectProposal → resolveDisruption. No mutation.

4. PERSONA BEHAVIOR AS NUMERIC PARAMETERS:
   Rather than using LLM prompts to simulate personality, each persona
   is a set of numeric parameters (conflict_level, fairness_sensitivity, etc.)
   that feed into deterministic scoring formulas. This allows:
   - Precise control over behavior
   - Statistical analysis via Monte Carlo
   - No LLM cost or latency

5. LAYERED ARCHITECTURE:
   The original 950-line route.ts was refactored into:
   - 15-line HTTP adapter (route.ts)
   - Orchestrator for routing
   - 7 standalone handlers
   - Shared state tracking
   - Isolated LLM client
   Each layer has a single responsibility and is independently testable.

6. PERSONALIZED DAY SUMMARIES:
   The explanation engine generates different text for each parent.
   "Sat Mar 14 | Emma & Jake with you." vs
   "Sat Mar 14 | Emma & Jake with Mom."
   This required passing a recipient parameter through the call chain.

7. SCHEDULE PREVIEW TRIGGERED BY MESSAGE TEXT:
   The inline calendar preview in PhoneSimulator.tsx is triggered by
   detecting "schedule is now created" in a system message. This means
   any code path that should show the calendar must include that exact
   phrase in its message text.

8. CROSS-PARENT NOTIFICATION ON COMPLETION:
   When Parent A completes onboarding via manual chat (send handler),
   Parent B receives two messages:
   - "Your family schedule is now created!" (triggers calendar)
   - "[Mom] has invited you to ADCP..." (context)
   These are separate messages so the trigger phrase is isolated.
