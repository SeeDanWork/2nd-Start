1. ARCHITECTURE OVERVIEW
================================================================================

The conversation system follows a strict layered architecture:

  HTTP Route (app/api/simulate/route.ts)
    |
    v
  Orchestrator (lib/conversation/orchestrator.ts)
    |
    v
  Action Handlers (lib/conversation/handlers/*.ts)
    |
    +---> Behavior Engine (lib/behavior-engine.ts)
    +---> Disruption Engine (lib/disruption-engine.ts)
    +---> Message Router (lib/message-router.ts)
    +---> Explanation Engine (lib/explanation-engine.ts)
    +---> Operational Messages (lib/operational-messages.ts)
    +---> Schedule Generator (lib/schedule-generator.ts)
    +---> Proposal Generator (lib/proposal-generator.ts)
    +---> LLM Client (lib/llm/client.ts)  [optional, fire-and-forget]

KEY DESIGN PRINCIPLES:
  - Deterministic first: All core logic runs without LLM. LLM is enrichment only.
  - Asymmetric messaging: Each parent gets different messages based on their role.
  - Immutable state transitions: Disruption state machine uses pure functions.
  - Personalized output: Day summaries use "you" for the recipient parent.
  - No mirroring: The system never sends the same message to both parents.


================================================================================
2. ORCHESTRATOR (Entry Point)
================================================================================

File: lib/conversation/orchestrator.ts

The orchestrator is the single entry point for all simulation actions. It:
  1. Looks up the scenario by ID
  2. Routes to the correct handler based on action type
  3. Normalizes errors into HTTP status codes
  4. Returns a uniform { status, data } response

ACTIONS (7 total):
  - connect         → Welcome message when parent first connects
  - send            → Process a user-typed message
  - auto_respond    → Persona-driven automatic response for one parent
  - simulate_pair   → Both parents respond simultaneously
  - run_setup       → Full automated onboarding for both parents
  - step_day        → Advance simulation by N days
  - inject_disruption → Trigger a disruption scenario

ERROR HANDLING:
  - 404: Scenario not found
  - 400: Validation errors (missing persona, invalid action)
  - 409: Duplicate disruption (same type + parent + day)
  - 500: Unexpected errors

Interface:
  orchestrate({ scenarioId, phone, action, body }) -> { status, data }

3. ACTION HANDLERS
================================================================================

All handlers live in lib/conversation/handlers/. Each is a standalone function
with typed inputs and outputs. The barrel export is in handlers/index.ts.

--- 3.1 CONNECT (connect.ts) ---

Purpose: First-time connection for a parent.

Logic:
  - If Parent A connects → standard onboarding welcome
  - If Parent B connects AND schedule already exists → contextual message
    ("Your co-parent has already set up your family schedule...")
  - If Parent B connects AND no schedule → standard onboarding welcome
  - Pushes the welcome message into the parent's message array
  - Returns schedule in response (so UI picks it up if schedule exists)

--- 3.2 SEND (send.ts) ---

Purpose: Process a user-typed message through the onboarding flow.

Logic:
  1. LLM enrichment (non-blocking): classifyIntent + extractEntities
  2. Append user message to parent's message array
  3. Build answered topics from message history
  4. Generate synthetic system response based on answered topics
  5. Append system response to message array
  6. If onboarding just completed (response matches "schedule is now created"):
     a. Generate schedule from config
     b. Push "schedule created" message to the OTHER parent
     c. Push "invite" message to the OTHER parent (separate message)

Cross-parent notification on completion:
  - Message 1: "Your family schedule is now created! ..."
  - Message 2: "[ParentLabel] has invited you to ADCP ..."
  These are separate messages so the "schedule created" text triggers
  the inline calendar preview in the UI.

--- 3.3 AUTO-RESPOND (auto-respond.ts) ---

Purpose: Generate a single persona-driven response for one parent.

Logic:
  1. Find the persona assigned to this parent
  2. Look up interaction archetype for the persona pair
  3. Analyze last system message to determine context (proposal/disruption/other)
  4. If proposal context → evaluateProposalWithArchetype → decision
  5. If disruption context → evaluateDisruption → decision
  6. Generate persona-appropriate message text
  7. LLM enrichment (classifyIntent on the generated text)
  8. Push user message + system response to message array

Decision outcomes drive message style:
  - accept → proposal_response message
  - counter → counter_text or swap_request message
  - reject → complaint message
  - ignore → "(no response)"

--- 3.4 SIMULATE PAIR (simulate-pair.ts) ---

Purpose: Both parents respond simultaneously in one action.

Logic:
  1. Both personas are required
  2. Look up interaction archetype
  3. For each parent: classify context (proposal/disruption/onboarding)
  4. If still onboarding → generateReactiveAnswer (topic-based)
  5. If post-onboarding → generatePersonaMessage (context-based)
  6. Evaluate decisions for both parents
  7. Build resolution paths based on decision combinations:
     - Both accept → "Mutual Agreement" (0.9 probability)
     - One accept, one resist → "One-Sided Accept" + "Escalation"
     - Counter → "Counter-Proposal" + "Mediation Trigger"
     - Reject → "Deadlock" + "Escalation to Default"
     - Ignore → "Timeout Auto-Resolve"
  8. Inject scenario-specific response patterns

--- 3.5 RUN SETUP (run-setup.ts) ---

Purpose: Complete automated onboarding for both parents.

Logic — Parent A:
  1. Generate welcome message
  2. Loop (max 15 turns):
     a. Check if onboarding complete
     b. generateCompoundAnswer: classify system message, generate answer
     c. Track answered topics
     d. LLM classify intent (fire and forget)
     e. Generate next synthetic system response
     f. Stuck detection: if same answer repeated twice, break

Logic — Parent B:
  1. Push initial join message
  2. generateParentBJoinResponse → "START"
  3. Loop (max 10 turns):
     a. Check if onboarding complete
     b. generateCompoundAnswer for each system question
     c. Track answered topics
     d. Stuck detection
  4. GUARANTEE: If Parent B didn't reach completion, push:
     - "Your family schedule is now created!" (triggers calendar preview)
     - "[ParentA label] has invited you to ADCP..." (context message)

Final steps:
  - Generate schedule from config
  - Set status to 'simulating'

--- 3.6 STEP DAY (step-day.ts) ---

Purpose: Advance simulation by N days, generating operational messages.

Logic per day:
  1. Get operational message for each parent (personalized by recipient)
  2. Check fairness alert (every 7 days)
  3. Check friction ahead (every 7 days)
  4. If all silent → generate personalized day summaries:
     - Parent A gets: buildDaySummaryExplanation(..., 'parent_a')
     - Parent B gets: buildDaySummaryExplanation(..., 'parent_b')
     - These produce different text ("you" vs parent label)
  5. If operational messages exist → push different texts to each parent

--- 3.7 INJECT DISRUPTION (inject-disruption.ts) ---

Purpose: Trigger a disruption scenario with full mediation flow.

See Section 8 (Disruption Flow) for detailed step-by-step.

4. CONVERSATION STATE TRACKING
================================================================================

File: lib/conversation/state.ts

Pure functions for tracking conversation progress. No side effects.

CONVERSATION PHASES:
  not_started       → No messages from either parent
  onboarding        → Messages exist but no schedule
  onboarding_complete → Schedule exists but currentDay = 0
  simulating        → Schedule exists and currentDay > 0
  disruption_active → Any active disruption not in RESOLVED state
  completed         → Scenario marked completed

KEY FUNCTIONS:

  getConversationPhase(scenario) → ConversationPhase
    Derives phase from scenario state. Checks in priority order:
    completed > disruption_active > simulating > onboarding_complete > onboarding > not_started

  buildAnsweredTopics(messages) → Set<string>
    Scans message history for system→user pairs. For each system message
    followed by a user reply, classifies the system message topics and
    adds them to the answered set. Skips 'unknown' and 'confirmed'.

  resolveParentContext(scenario, phone) → ConversationState
    Returns { phase, answeredTopics, isParentA, messages } for the
    given parent's phone number.

  getLastSystemMessage(messages) → string
    Finds the most recent system message text.

  isOnboardingDone(messages) → boolean
    True if last system message is classified as 'confirmed'.

5. ONBOARDING FLOW (Synthetic System)
================================================================================

File: lib/behavior-engine.ts (lines 656-733)

The onboarding flow is a fixed sequence of steps. Each step has:
  - step name
  - askTopics: which topics this step covers
  - systemMessage: function that generates the system prompt

STEP SEQUENCE:
  1. welcome          → asks: children_count
  2. arrangement      → asks: arrangement
  3. split_distance   → asks: split, distance
  4. locked_weekends  → asks: locked_days, weekends
  5. exchange_frustrations → asks: exchange, frustrations
  6. partner_phone    → asks: phone
  7. confirm          → asks: confirm (shows summary for approval)
  8. complete         → asks: confirmed (terminal — schedule created message)

generateSyntheticSystemResponse(config, answeredTopics):
  Iterates through ONBOARDING_FLOW. Returns the system message for the
  first step that has at least one unanswered topic. If all answered,
  returns the completion message.

TOPIC CLASSIFICATION (classifySystemMessage):
  Uses keyword matching on the system message text to detect which
  onboarding topic is being asked about. Patterns:
    greeting    → "welcome", "get started"
    children_count → "how many children"
    arrangement → "custody", "arrangement"
    distance    → "how far", "miles"
    phone       → "phone number"
    confirmed   → "schedule is now created"
    confirm     → "does this look right"
    locked_days → "specific nights"
    etc.

classifyAllTopics(text):
  Returns ALL matching topics (not just first). Used for compound questions
  that cover multiple topics. Terminal states ('confirmed') are exclusive.

REACTIVE ANSWER GENERATION:
  generateReactiveAnswer(topic, persona, config, answeredSet):
    Generates a persona-appropriate answer for a single topic using config data.
    Returns empty string if topic already answered (unless topic is 'confirm').

  generateCompoundAnswer(systemMessage, persona, config, answeredSet):
    Classifies all topics in the system message, generates answers for
    unanswered ones, combines them. If all topics answered, returns 'confirm'.

6. BEHAVIOR ENGINE (Persona-Driven Decisions)
================================================================================

File: lib/behavior-engine.ts

DECISION TYPES: accept | reject | counter | ignore | delay

PROPOSAL EVALUATION — evaluateProposal(persona, fairnessDeviation, ...):
  Scoring formula:
    score = proposal_acceptance_bias
          - (fairnessDeviation/100) * (fairness_sensitivity/5)  [if dev > 5]
          - (schedule_rigidity/5) * 0.2                         [if transitions increase]
          + 0.15                                                [if disruption + low conflict]
          - (conflict_level/5) * 0.15
          + (logistics_tolerance/5) * 0.1

  Decision thresholds:
    roll < score        → accept  (confidence: score)
    roll < score + 0.3  → counter (if gaming_probability > threshold)
    else                → reject  (confidence: 1 - score)

  Avoidant override: 40% chance of 'ignore' for very_slow personas

DISRUPTION EVALUATION — evaluateDisruption(persona, event, currentDay):
  Uses conflict_level and logistics_tolerance to determine response.
  Lower conflict + higher tolerance → more likely to accept.

ARCHETYPE-AWARE EVALUATION — evaluateProposalWithArchetype(persona, archetype, ...):
  If an archetype exists for the persona pair, adjusts scoring:
    - High conflict_probability → lower acceptance
    - Archetype's typical resolution patterns influence thresholds

PERSONA MESSAGE GENERATION — generatePersonaMessage(persona, situation, context):
  Situations: greeting, complaint, proposal_response, swap_request,
              disruption_report, negotiation, followup
  Uses persona's behavior traits to select from message templates.
  High fairness_sensitivity → fairness-focused messages.
  High conflict_level → resistant/combative messages.

RESPONSE TIMING — getResponseDelay(persona):
  fast:      5-30 seconds
  medium:    60-300 seconds (1-5 min)
  slow:      600-1800 seconds (10-30 min)
  very_slow: 1800-7200 seconds (30 min - 2 hours)

7. LLM ISOLATION LAYER
================================================================================

File: lib/llm/client.ts

ALL external LLM calls go through this single module. Three functions:

  classifyIntent(input, context?) → IntentResult | null
    POST /llm/classify-intent
    Returns { intent, confidence } or null on failure.

  extractEntities(input, context?) → EntityResult | null
    POST /llm/extract-entities
    Returns entity map or null on failure.

  generateExplanation(input, context?) → string | null
    POST /llm/generate-explanation
    Returns explanation text or null on failure.

FAILURE MODE: All functions return null on any error (network, timeout,
non-200 response). The system continues with deterministic-only processing.
LLM calls are ENRICHMENT, never blocking.

Environment: LLM_ROUTER_URL (default: http://localhost:3100)

8. DISRUPTION FLOW (Mediation-Based)
================================================================================

File: lib/disruption-engine.ts (state machine)
File: lib/conversation/handlers/inject-disruption.ts (handler)
File: lib/message-router.ts (asymmetric messages)
File: lib/proposal-generator.ts (coverage options)

STATE MACHINE (8 states, immutable transitions):

  DISRUPTION_REPORTED → DURATION_ASKED → COVERAGE_REQUESTED
    → PROPOSALS_GENERATED → PARENT_RESPONSE_PENDING
    → RESOLUTION_APPLIED → FOLLOWUP_PENDING → RESOLVED

  Or short path: ... → RESOLUTION_APPLIED → RESOLVED (today_only duration)

DISRUPTION TYPES:
  parent_sick, child_sick, work_emergency, transport_failure,
  school_closure, family_emergency, schedule_conflict, other

DURATION OPTIONS:
  today_only, 2_3_days, week, unknown

COVERAGE TYPES:
  full_custody_transfer, partial_day_coverage,
  transport_assistance, temporary_logistics_swap

FULL INJECTION FLOW (inject-disruption.ts):

  Step 1: REPORTER gets disruption confirmation
    - "Event recorded: [event label]"
    - "[Children] scheduled with [you/other] today"
    - Options: Request coverage / Manage myself

  Step 2: REPORTER auto-sends disruption report
    - Persona-driven message: generatePersonaMessage(persona, 'disruption_report')

  Step 3: REPORTER gets duration question
    - "How long? 1. Today only  2. 2-3 days  3. Week  4. Not sure"

  Step 4: Auto-simulate duration answer
    - difficulty <= 2 → today_only
    - difficulty <= 3 → 2_3_days
    - difficulty > 3  → unknown

  Step 5: OTHER PARENT gets coverage request
    - "Coverage request from [reporter]. Reports: [event]. Duration: [X]"

  Step 6: Generate proposal bundle (proposal-generator.ts)
    - Full coverage transfer (all affected days)
    - Partial day coverage (if <= 2 affected days)
    - Shortened coverage (ceil(affected/2) days, if > 2 affected days)
    - Each option includes fairness impact, transition impact, compensation days

  Step 7: OTHER PARENT gets proposal options

  Step 8: OTHER PARENT auto-responds (evaluateDisruption)
    - accept → selects first option, both get resolution trace
    - reject → decline all, both get decline confirmation
    - counter → selects partial option if available, else declines

  Step 9: Follow-up or resolve
    - today_only → RESOLVED immediately
    - longer duration → FOLLOWUP_PENDING

IDEMPOTENCY: isDuplicateDisruption() prevents same parent + event type
on the same day. Returns 409 if duplicate.

DISRUPTION TYPE CLASSIFICATION (classifyDisruptionType):
  Regex-based keyword matching on scenario description.
  IMPORTANT: child_sick patterns checked BEFORE parent_sick patterns
  because "child sick" contains "sick" which would match parent_sick first.

9. MESSAGE ROUTING (Asymmetric Messaging)
================================================================================

File: lib/message-router.ts

Core principle: Each parent receives contextually appropriate messages
based on their role (reporter vs other parent). Never mirrors.

MESSAGE TYPES:
  DISRUPTION_REPORT_CONFIRMATION → Reporter only
  DURATION_QUESTION              → Reporter only
  COVERAGE_REQUEST               → Other parent only
  PROPOSAL_BUNDLE                → Other parent only
  PROPOSAL_SELECTED              → Both (different text)
  CALCULATION_TRACE              → Both (same metrics, different framing)
  DECLINE_CONFIRMATION           → Both (different text)
  MANAGE_SELF_CONFIRMATION       → Reporter only
  FOLLOWUP_CHECK                 → Reporter only

Each function returns a RoutedMessage with:
  { type, recipient, text }

Or for dual-send messages:
  { reporter: RoutedMessage, other: RoutedMessage }

10. OPERATIONAL MESSAGES (Day-to-Day)
================================================================================

File: lib/operational-messages.ts

During stable periods (no disruptions), the system generates structured
operational messages. Core principle: "silent unless useful."

MESSAGE TYPES:
  SILENT                  → No message (most days)
  DAILY_SUMMARY           → Brief day status
  RESPONSIBILITY_REMINDER → Transition day reminder
  LOGISTICS_CLARIFICATION → Exchange logistics details
  WEEKLY_OVERVIEW         → End-of-week summary (day 7, 14, ...)
  FAIRNESS_ALERT          → Fairness drift > threshold
  LONG_STRETCH_ALERT      → One parent has had child 5+ consecutive nights
  MONTHLY_SUMMARY         → End-of-month report
  OPTIMIZATION_PROMPT     → Suggestion for schedule improvement
  HOLIDAY_PLANNING        → Upcoming holiday coordination
  FRICTION_DETECTION      → Predicted conflict point ahead

getOperationalMessage(config, schedule, simDay, recipient):
  Returns personalized message for the given parent.
  Message priority: Weekly > Monthly > Transition > Fairness > Silent.

SCHEDULE METRICS (computeScheduleMetrics):
  Computed over a rolling window (default 56 days / 8 weeks):
  - fairnessBalance: { parentA nights, parentB nights }
  - transitionsThisWeek: count of custody switches
  - avgBlockLength: average consecutive nights with same parent
  - stabilityScore: strong (few transitions) / moderate / weak
  - schoolNightExchanges: transitions on Sun-Thu nights
  - longestStreak: longest consecutive block for either parent

11. EXPLANATION ENGINE (Deterministic Traces)
================================================================================

File: lib/explanation-engine.ts

Generates structured explanations from schedule metrics. No LLM reasoning.
Template + metrics → explanation.

EXPLANATION OBJECT:
  {
    change: { summary, detail }
    factors: [{ label, status: improved|unchanged|degraded, value }]
    comparison: { before: MetricSnapshot, after: MetricSnapshot }
    scores: { fairness_penalty, transition_penalty, routine_penalty,
              fragmentation_penalty, total_score }
  }

THREE FORMAT LEVELS:
  Level 1 (formatLevel1): One-line summary
    "Changed custody. Improves fairness balance."

  Level 2 (formatLevel2): Factor breakdown
    "[+] Fairness balance: 28/28"
    "[=] Transitions/week: 2"
    "[-] Avg block length: 3 nights"

  Level 3 (formatLevel3): Debug score components
    "fairness_penalty = -2"
    "transition_penalty = 0"
    "total_score = -1"

PERSONALIZED DAY SUMMARIES (buildDaySummaryExplanation):
  Accepts optional recipient parameter ('parent_a' | 'parent_b').
  When provided:
    - Uses "you" for the recipient's custody days
    - Uses the other parent's name for their days
    - "Next exchange: Fri Mar 20 to you" vs "to Dad"
  When not provided:
    - Uses parent labels for both

12. DATA FLOW DIAGRAMS
================================================================================

--- Manual Chat Flow (User Types in Phone) ---

  User types in Mom's phone
    |
    v
  POST /api/simulate { action: 'send', phone: '+1111', body: 'text' }
    |
    v
  orchestrate() → handleSend()
    |
    +---> classifyIntent(body)         [LLM, non-blocking]
    +---> extractEntities(body)        [LLM, non-blocking]
    |
    +---> messagesA.push(user message)
    +---> buildAnsweredTopics(messagesA)
    +---> generateSyntheticSystemResponse(config, topics)
    +---> messagesA.push(system response)
    |
    +---> if onboarding complete:
    |       generateSchedule(config)
    |       messagesB.push("Schedule created!")
    |       messagesB.push("[Mom] invited you to ADCP")
    |
    v
  Response: { messagesA, messagesB, schedule, logs }

--- Disruption Flow ---

  POST /api/simulate { action: 'inject_disruption', body: 'child_sick_exchange' }
    |
    v
  handleInjectDisruption()
    |
    +---> REPORTER: confirmation + report + duration question
    +---> Auto-determine duration
    +---> OTHER: coverage request
    +---> generateProposalBundle() → options
    +---> OTHER: proposal options
    +---> evaluateDisruption(otherPersona) → decision
    +---> Route accept/reject/counter:
    |       Accept: selectProposal → resolution trace to both
    |       Reject: declineAll → decline message to both
    |       Counter: select partial → partial resolution
    +---> FOLLOWUP_PENDING or RESOLVED
    |
    v
  Response: { messagesA, messagesB, schedule, activeDisruption }

--- Day Step Flow ---

  POST /api/simulate { action: 'step_day', body: 7 }
    |
    v
  handleStepDay() — loop 7 times:
    |
    +---> getOperationalMessage(config, schedule, day, 'parent_a')
    +---> getOperationalMessage(config, schedule, day, 'parent_b')
    +---> checkFairnessAlert() [every 7 days]
    +---> checkFrictionAhead() [every 7 days]
    |
    +---> If silent:
    |       buildDaySummaryExplanation(config, schedule, day, 'parent_a') → A
    |       buildDaySummaryExplanation(config, schedule, day, 'parent_b') → B
    +---> If operational:
    |       Different text for each parent
    |
    v
  Response: { days[], totalDaysRun, messagesA, messagesB, schedule }
